## UNIGINE Engine Key Concepts

### Coordinate System

UNIGINE uses a **right-handed** coordinate system:
- **+X** = Right
- **+Y** = Forward
- **+Z** = Up

Positive rotation is **counterclockwise** (right-hand rule).

Units are **meters**.

**Camera forward direction:** For Players (cameras), forward is **-Z in local space**, accessed via `GetWorldDirection(MathLib.AXIS.NZ)` (C#) or `getWorldDirection(Math::AXIS_NZ)` (C++). This is different from regular nodes where forward is +Y.

### World and Nodes

A scene is called a **World** (`.world` file). Everything in a world is a **Node**.

Node types include:
- **Objects**: visible geometry — `ObjectMeshStatic`, `ObjectMeshSkinned`, `ObjectMeshDynamic`, `ObjectTerrain`, `ObjectWater`, etc.
- **Lights**: `LightWorld` (directional), `LightOmni` (point), `LightProj` (spot)
- **Players**: cameras — `PlayerDummy`, `PlayerSpectator`, `PlayerPersecutor`, `PlayerActor`
- **Utility**: `NodeDummy`, `NodeReference` (prefab), `NodeLayer`, `NodeTrigger`

Nodes have a parent-child hierarchy. Child transforms are relative to parent.

### Surfaces

A mesh can have **multiple Surfaces** — each surface is a sub-mesh with its own material. Surfaces are set at import time and cannot be added/removed at runtime. Per-surface settings include: material, LOD distances, collision flag, shadow casting, visibility masks.

### Materials

Materials form a **hierarchy**: base materials (read-only, define shaders) → user materials (inherit and override parameters). The standard PBR material is `mesh_base`.

Normal maps use **DirectX convention** (Y-).

### Components

Components add gameplay logic to nodes. Two systems available:

**C# components** MUST use `public partial class`:
```csharp
public partial class MyComponent : Component
{
    public float speed = 5.0f;  // exposed in Editor

    private void Init() { }
    private void Update() { }
    private void Shutdown() { }
}
```

**Do NOT use `[Component(PropertyGuid = "...")]` attribute.** GUIDs are project-specific and auto-generated by the engine. Never copy GUIDs from samples or other projects.

**Do NOT omit `partial` keyword.** Without `partial`, the C# source generator (SCG) cannot generate the property file and compilation will fail with SCG001.

**C++ components** inherit from `ComponentBase`:
```cpp
class MyComponent : public ComponentBase
{
public:
    COMPONENT_DEFINE(MyComponent, ComponentBase);
    COMPONENT_INIT(init);
    COMPONENT_UPDATE(update);
    COMPONENT_SHUTDOWN(shutdown);

    PROP_PARAM(Float, speed, 5.0f);

    void init();
    void update();
    void shutdown();
};
```

Component lifecycle: `Init()` → `UpdateAsyncThread()` → `UpdateSyncThread()` → `Update()` → `PostUpdate()` → `UpdatePhysics()` → `Swap()` → `Shutdown()`.

### Player (Camera) Setup

To make a camera work, you MUST set `Game.Player`:
```csharp
private void Init()
{
    Game.Player = node as Player;  // required — without this, black screen
}
```

Without `Game.Player = ...`, the viewport will be a black screen.

### Execution Sequence

The application has two logic layers:
- **SystemLogic** (`AppSystemLogic`): lives for the entire application lifetime
- **WorldLogic** (`AppWorldLogic`): lives while a world is loaded

Per-frame order: SystemLogic.Update → Components.Update → Physics → Components.PostUpdate → Render → SystemLogic.PostUpdate.

Physics runs at a **fixed FPS** (default 60), independent of rendering FPS.

### Delta Time

Frame-rate independent movement uses **`Game.IFps`** (C#) or **`Game::getIFps()`** (C++):
```csharp
node.WorldPosition += node.GetWorldDirection(MathLib.AXIS.Y) * speed * Game.IFps;
```

This is the inverse of frame time (1/fps).

### Direction Vectors

```csharp
// For regular nodes:
vec3 forward = node.GetWorldDirection(MathLib.AXIS.Y);  // +Y = forward
vec3 right   = node.GetWorldDirection(MathLib.AXIS.X);  // +X = right
vec3 up      = node.GetWorldDirection(MathLib.AXIS.Z);  // +Z = up

// For Players (cameras) — forward is -Z in local space:
vec3 cameraForward = player.GetWorldDirection(MathLib.AXIS.NZ);
```

### Node Lifetime

Each node has a lifetime mode:
- **WORLD** (default): deleted when the world is closed
- **ENGINE**: persists across world switches, deleted on engine shutdown
- **MANUAL**: you must delete it yourself via `node.DeleteLater()`

### Bit Masks

UNIGINE uses **32-bit masks** instead of tags/layers for filtering:
- Viewport Mask — which objects render
- Collision Mask — which objects collide
- Shadow Mask — which objects cast shadows
- **Intersection Mask** — which objects participate in raycasts (`World.GetIntersection`)

Two objects interact when their masks share **at least one matching bit** (bitwise AND > 0).

**Primitives and programmatically created objects may have intersection_mask = 0 by default.** If `World.GetIntersection` returns null for an object you expect to hit, check its intersection mask:
```csharp
(node as Unigine.Object)?.SetIntersectionMask(0xFFFFFFFF, 0);  // SetIntersectionMask(mask, surface)
```

### Physics

Bodies describe physical behavior; Shapes define collision volumes.

- **Body types**: Rigid, Dummy (static), Ragdoll, Fracture, Cloth, Rope, Water
- **Shape types**: Box, Sphere, Capsule, Cylinder, Convex Hull
- **Joint types**: Fixed, Hinge, Ball, Prismatic, Cylindrical, Wheel, Suspension

Collision detection: shape-vs-shape and shape-vs-surface (enable "Collision" flag on surface).

**Do not scale physics objects at runtime** — reimport with the correct scale instead.

**Static physics floor/walls:** To make a static surface solid (e.g. arena floor), add `BodyRigid` with `Immovable = true` and a `ShapeBox` matching the geometry. Without a body+shape, players and objects will fall through.

**Physics shape sizes must match visual geometry.** When creating physics shapes in code, read the object's dimensions at runtime (e.g. via bounding box) or use shared constants. Do NOT hardcode sizes independently in two places — they will desynchronize when geometry changes.

### Intersections (Raycasting)

UNIGINE calls raycasts **Intersections**:
```csharp
// World intersection (visual geometry)
ObjectIntersection hit = new ObjectIntersection();
Unigine.Object obj = World.GetIntersection(p0, p1, mask, hit);

// Physics intersection (collision shapes)
PhysicsIntersection hit = new PhysicsIntersection();
Unigine.Object obj = Physics.GetIntersection(p0, p1, mask, hit);
```

### Rendering

- **Deferred** pipeline for opaque geometry
- **Forward** pipeline for transparent geometry
- Opaque/transparent is determined by material Transparency Preset
- LOD system: discrete (hard switch) or smooth (alpha fade) by distance

### Node References (Prefabs)

Save a node hierarchy to a `.node` file, then instance it via `NodeReference`:
```csharp
NodeReference nodeRef = new NodeReference("my_prefab.node");
```

### Finding Nodes

```csharp
Node n = World.GetNodeByName("my_node");           // by name
Node n = World.GetNodeById(id);                     // by ID
List<Node> nodes = node.GetChildren();              // children
Node child = node.FindNode("child_name", 1);          // recursive search (int, not bool)
```

## C++ vs C# API Differences

**Method access patterns differ between languages.** Always verify in the correct language docs.

| Concept | C++ | C# |
|---------|-----|-----|
| Set position | `node->setWorldPosition(pos)` | `node.WorldPosition = pos` (property) |
| Check valid | `node.isValid()` (Ptr<> method) | No direct equivalent; use `node != null` |
| Delta time | `Game::getIFps()` | `Game.IFps` (property) |
| Delete node | `node.deleteLater()` (Ptr<> method) | `node.DeleteLater()` |
| Math PI | `Math::PI` | `MathLib.PI` (also `MathLib.PI2`, `MathLib.DEG2RAD`) |
| Key names | `KEY_1` | `Input.KEY.DIGIT_1` (not `KEY.ONE`) |

## GUI (Widgets)

### Visibility
- **`Widget.Hidden = true/false`** — show/hide a widget. This is what you want for toggling UI elements.
- **`Widget.Enabled = false`** does NOT hide the widget — it grays it out (disables interaction but stays visible).

### Namespace conflicts
When using `using Unigine;`, some .NET types conflict:
- **`Random`** — use `System.Random` explicitly (conflicts with `Unigine.Random`)
- **`Object`** — use `Unigine.Object` explicitly (conflicts with `System.Object`)

## Double vs Float Precision

UNIGINE supports two coordinate precision modes:

| | Float (single) | Double |
|---|---|---|
| Precision | 32-bit | 64-bit |
| World size | up to ~10 km from origin | virtually unlimited |
| Performance | faster | ~10-15% slower |
| Position types | `vec3`, `mat4` | `dvec3`, `dmat4` |
| Use case | small/medium scenes | large worlds, GIS, space |
| Preprocessor | — | `UNIGINE_DOUBLE` defined |
| DLLs | `Unigine_x64.dll` | `Unigine_double_x64.dll` |

To write code that works with both builds, use capital-letter types: `Scalar` (float or double), `Vec3` (vec3 or dvec3), `Mat4` (mat4 or dmat4).

In double-precision projects, all coordinate-related APIs (`setPosition`, `getWorldTransform`, etc.) use `dvec3`/`dmat4` instead of `vec3`/`mat4`.

| | C# | C++ |
|---|---|---|
| Methods | `PascalCase` | `camelCase` |
| Properties | `PascalCase` | `getCamelCase()` / `setCamelCase()` |
| Enums | `UPPER_CASE` | `UPPER_CASE` |

| Namespace | `Unigine` | `Unigine` |
| Casting | `node as ObjectMeshStatic` | `checked_ptr_cast<ObjectMeshStatic>(node)` |

## C++ Smart Pointers (Ptr<>)

In C++, all UNIGINE objects are managed through `Ptr<T>` smart pointers (`NodePtr`, `ObjectPtr`, `MaterialPtr`, etc.).

**Critical distinction:** `Ptr<T>` has its own methods that are NOT methods of `T`:
```cpp
NodePtr node = World::getNodeByName("my_node");

node->setWorldPosition(pos);  // → calls Node::setWorldPosition (operator->)
node.deleteLater();            // → calls Ptr<Node>::deleteLater (dot operator!)
node.get();                    // → returns raw Node* pointer
node.isNull();                 // → checks if pointer is null
```

`deleteLater()` is a method of **Ptr<>**, not of **Node**. Using `node->deleteLater()` will not compile.

**Inheritance:** `Object` inherits from `Node`, so `ObjectPtr` implicitly converts to `NodePtr`. There is no `obj->getNode()` method — the object IS a node:
```cpp
ObjectMeshStaticPtr obj = ObjectMeshStatic::create("mesh.mesh");
NodePtr node = obj;  // implicit conversion, no cast needed
```

## C++ vs C# Project Differences

| Aspect | C++ | C# |
|---|---|---|
| Component base class | `ComponentBase` | `Component` |
| Component location | `source/` (compiled with project) | `data/` (compiled by Editor automatically) |
| Build step | Manual (CMake/VS, rebuild required) | Automatic (Editor compiles on load) |
| Play button in Editor | **Not available** — run via SDK Browser or `launch_debug.bat` | Available in Editor |
| Runtime binary | `bin/<project>_x64d.exe` (separate from Editor) | Editor runs components directly |
| Property parameters | `PROP_PARAM(Type, name, default)` macro | Public fields auto-exposed |
| Component registration | `REGISTER_COMPONENT(Class)` in .cpp | Automatic |

**C++ Editor has no Play button.** To test C++ code, build the project and run the runtime binary. The Editor (`Editor_double_x64.exe`) and the runtime (`<project>_x64d.exe`) are separate executables — Editor does not execute user C++ code.

### After Writing a Component

After writing or modifying component code (.cs or .cpp):

**Step 1 — Build (both C# and C++).** Propose building the project to the user to check for compilation errors. Wait for user approval, then run the build (dotnet build for C#, CMake/Visual Studio .sln for C++). Fix any errors before proceeding.

**Step 2 — Make components available in the Editor:**
- **C#:** Ask the user to make the Editor window active (click on it or Alt-Tab to it). The Editor automatically detects changed .cs files and recompiles them — after that, the new component appears in the component list.
- **C++:** Ask the user to build (if not already built) and run the application. C++ components are initialized at application startup together with the component system — the Editor does not execute user C++ code, so components only become available after the application has been launched at least once.

Components do not appear in the Editor until these steps are completed.

## Common Pitfalls

### Double Precision Types
In double-precision projects, use the correct types for world-space operations:
- `WorldBoundSphere` (not `BoundSphere`) — takes `dvec3` + `double`
- `WorldBoundBox` (not `BoundBox`) — takes `dvec3`
- `BoundSphere`/`BoundBox` use `vec3` (float) — only for local-space

### Intersection Types
Both `ObjectIntersection` and `WorldIntersection` exist in C++ and C#. They serve different purposes:
- **`ObjectIntersection`** — stores surface index, triangle index, and local coordinates
- **`WorldIntersection`** — stores world-space point, normal, triangle index, and surface index

```csharp
// C# — World.GetIntersection with ObjectIntersection
ObjectIntersection hit = new ObjectIntersection();
Unigine.Object obj = World.GetIntersection(p0, p1, mask, hit);
```
```cpp
// C++ — World::getIntersection with WorldIntersectionPtr
WorldIntersectionPtr hit = WorldIntersection::create();
ObjectPtr obj = World::getIntersection(p0, p1, mask, hit);
```

### Cross Product and Coordinate System
In UNIGINE's right-handed system: `cross(forward, up) = right`, i.e. `cross(+Y, +Z) = +X`.
```cpp
vec3 forward = node->getWorldDirection(AXIS_Y);
vec3 right = cross(forward, vec3_up);  // correct: gives +X
// NOT: cross(vec3_up, forward) — this gives LEFT (-X)
```

### Physics
- **BodyDummy** is a static body for attaching joints — it is NOT a collision-only body
- **PhysicalTrigger** (not WorldTrigger) is the zone-trigger node for enter/leave callbacks
- **CCD** (Continuous Collision Detection) is enabled per **Shape** (flag `Continuous`), not per Body. Only Sphere and Capsule shapes support CCD
- Physics always runs at fixed FPS. The two update modes are **Before Rendering** (synchronous) and **Async Rendering** (parallel, default) — not "fixed vs variable timestep"

### Bit Masks (Extended List)
UNIGINE has 10+ mask systems, not just the basic 4:
- Viewport, Shadow, Collision, Physics Intersection (basic)
- Reflection, Decal, Field, Sound Source, Physical, Navigation/Pathfinding, Exclusion (extended)

### Materials
- Base materials can be **user-created** (via code or Material Editor graph), not only engine-built-in
- Texture resolution must be **divisible by a power of 2** (e.g. 768 is valid), not necessarily a power of 2 itself
- To find a material by name in code: use `Materials.FindManualMaterial()` (NOT `FindMaterial` — does not exist)
