# Fixed-Wing Template - Background Traffic Simulation


The template contains the framework simulating **background vehicle movement along roads**. The simulation is intentionally approximate rather than physically accurate, and is designed for use cases where traffic is observed from a distance - for example, from a pilot’s cockpit.


![](../modules/traffic/intro.jpg)


The following key requirements have been taken into account for the traffic simulation:


- **Low performance cost** Minimal CPU calculations are used so that traffic simulation does not consume resources needed by more critical gameplay or simulation logic.
- **Synchronization support** The simulation must behave identically across multiple machines or across multiple instances of the application running on the same machine. For this reason, traffic simulation is integrated with the [Syncker](../../../code/plugins/syncker/index.md) synchronization system.


Traffic movement is based on a **graph** structure (created using [WorldSplineGraph](../../../objects/worlds/world_spline_graph/index.md) node) that forms the road network used by vehicles.


To simulate the traffic, the following components are used:


- **TrafficGraphController** - assigned to the World Spline Graph node, as it generates the traffic graph based on that node.

  - Provides access to the traffic configuration parameters.
  - Converts World Spline Graph to JSON.
  - Calculates current transformations of the vehicles within the car Mesh Cluster.
- **TrafficVehicleType** - assigned to the car Mesh Cluster, which is the child node of World Spline Graph.

  - Provides access to vehicle speed.
  - During the *update()* sets the current transforms of vehicles, calculated by *TrafficGraphController*.


## Implementation Overview


The background traffic simulation is configured as follows:


1. A [traffic graph is created](#create_graph) and appropriate [segmentation is configured](#graph_segmentation).
2. The graph is [converted to JSON](#generate_json) by using the corresponding *TrafficGraphController* component parameter.
3. The related traffic settings are adjusted.
4. A vehicle cluster is created, the mesh and materials are assigned to it, and settings are configured using the corresponding component.


## Creating a Graph


To simplify graph creation in the Editor, the WorldSplineGraph node is used. It allows placing control points and configuring road curvature visually.


![](../modules/traffic/wsg.jpg)


The native `*.spl` file generated by WorldSplineGraph contains excessive data, so the traffic system transforms it to a more lightweight `*.json` file that stores a simplified representation of the graph (an adjacency-based structure).


To generate the `*.json` file from an `*.spl` file, an output location is specified in the corresponding field of the *TrafficGraphController* component. The file is generated at runtime:


![](../modules/traffic/graph_settings_json_generation.png)

If you want to use a JSON file without any optimizations, disable the generation option and specify the path to your file:

### Graph Segmentation


During generation, each graph edge is subdivided into smaller sub-edges using a configurable **Segment split distance**. Edges are split into segments regardless of whether they are curved or straight.


## Adding Traffic


### Vehicle Movement Logic


Vehicles are not aware of the entire graph. Instead, they move in the following manner:


- A vehicle travels from vertex to vertex.
- When it reaches the target within a defined threshold, it begins turning toward the next vertex.
- Movement is continuous and approximate, rather than following an exact spline path.


This approach keeps computation lightweight while producing visually convincing results at a distance.


### Traffic Settings


The traffic system exposes several configuration parameters available in the *TrafficGraphController* component:


![](../modules/traffic/trafficgraphcontroller_component.png)


| Parameter | Description |
|---|---|
| Visibility Distance | The maximum distance at which vehicles are visible. |
| Spawn Distance Min | Vehicles are spawned only around the player and only at graph vertices within this distance. |
| Traffic Density | Defines traffic density as the number of vehicles per visible graph vertex. |
| Max Vehicle Count | Upper limit on the total number of vehicles. |
| Vehicle Reach Target Distance | The distance at which a vehicle considers a target vertex reached and begins steering toward the next one. This controls how early the vehicle starts turning. |
| Check Collision Distance | Distance at which a vehicle stops if another vehicle ahead is moving toward the same target vertex. |
| Max Speed (km/h) | Maximum possible velocity of a vehicle. |
| Start Timer Delay | Ensures traffic starts simultaneously across all synchronized instances, accounting for cases where a Slave instance loads significantly later than the Master. |
| Normalized Dot Threshold for Same Direction | When checking for a vehicle ahead, the system verifies that both vehicles are moving in the same direction using the dot product of their movement vectors. The value must be within [-1, 1]; higher values indicate a smaller angle between vectors and better directional alignment. |
| Road Center Offset | Controls lateral vehicle placement relative to the traffic graph. Vehicles are spawned with a sideways offset from the graph vertex, allowing them to occupy separate lanes instead of driving directly toward each other. |


### World Structure and Visual Representation


Vehicle visuals are managed using [Mesh Clusters](../../../objects/objects/mesh_cluster/index.md) through the following approach:


- Mesh Clusters are added as children of World Spline Graph, one for each type of vehicle. ![](../modules/traffic/clusters.png)
- The car mesh and material are set via Mesh Cluster.
- The *TrafficVehicleType* component is assigned to the Cluster node. It provides access to the vehicle speed setting. ![](../modules/traffic/trafficvehicletype.png)
- Within the *TrafficGraphController* component, references to available clusters are provided along with “raw” spawn probabilities. These probabilities are normalized internally. ![](../modules/traffic/spawn_settings.png) For example, if three vehicle types are provided with equal weights, each will have a spawn probability of **0.33**.
- Within the *TrafficGraphController* component, **Debug mode** is available. When enabled, the actual traffic graph and the target vertices currently assigned to vehicles are rendered by the Engine visualizer. ![](../modules/traffic/debug.png)


## Limitations


- The system works best when **FPS is stable** on both Master and Slave instances. If the Slave experiences frame drops, traffic may temporarily desynchronize, but should gradually re-sync over time.
- Vehicles do **not follow graph edges precisely**. They move directly from point to point without taking into account the turn curvature. Proper behavior requires careful tuning using: For example, a crossroads (two roads crossing perpendicularly) can be implemented in either of the following ways: | ![](../modules/traffic/point_crossroads.jpg) | **Simply placing points in the Editor** - quicker to implement, but requires additional parameter tuning to ensure vehicles navigate turns correctly. This approach is applied uniformly to all turns, which may be beneficial if turn geometry is relatively consistent, but less suitable when turn types vary significantly. | |---|---| | ![](../modules/traffic/spline_crossroads.jpg) | **Using spline curves for each lane** - offers greater control over vehicle movement. This method requires finer graph segmentation so that vehicle motion more accurately follows the curvature of the graph. |

  - Vehicle speed
  - Graph segmentation length
  - Placement of WorldSplineGraph control points
  - Turn angle logic (configured in code)
- If vehicle speed is too high, graph segmentation too coarse, and [Vehicle Reach Target Distance](#reach_target_distance) (*TrafficGraphController* component setting) too large, vehicles may overshoot sharp turns and begin circling around a single vertex. This can be resolved by:

  - Increasing graph segmentation density
  - Reducing vehicle speed
  - Increasing turn angle limits (configured in code)
